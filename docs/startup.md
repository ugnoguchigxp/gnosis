# Startup Improvement Plan

## 目的

`git clone` 直後の新規利用者が、過度な前提知識なしで Gnosis を起動し、最小限の価値を確認できる状態を作る。

この計画では、導入体験の改善を以下の順序で進める。

1. 最短導線を定義する
2. 判断が必要な箇所を減らす
3. セットアップを自動化する
4. 失敗時の自己診断を可能にする
5. fresh clone で継続的に壊れないようにする

## 現状の課題

- `README.md` が概念説明中心で、初回導入の最短手順が前面に出ていない
- `.env.example` に必須設定と任意設定が混在しており、どこまで埋めれば良いか判断しづらい
- `bun run monorepo:setup` が Python サービス準備のみで、導入全体の完了を保証しない
- DB、embedding、local-llm、monitor のどこまでが必須か明確ではない
- 導入成功を確認する単一コマンドがない
- fresh clone 環境で導入手順を継続検証する CI がない

## 目標

- 初回利用者が `README` 冒頭だけで最小起動まで到達できる
- 導入コマンドが「最小構成」と「フル構成」で分かれている
- 失敗時に、原因と次アクションがコマンド単位で返る
- `git clone` からの導入手順が CI で検証される
- 実装に不足パッケージがあれば追加し、運用しやすい構成で維持する
- 現時点では Windows を正式サポートしなくても、セットアップ設計は Windows 拡張を阻害しない

## 非目標

- 初回フェーズで local-llm や Monitor の体験を完全自動化すること
- すべてのクラウド LLM プロバイダ設定を同じ深さで整備すること
- 開発者向けの詳細ドキュメントを削ること

## 実装方針

- 最初に「何が必須か」を固定し、その後に自動化を載せる
- 1つの巨大セットアップではなく、失敗点が分離された段階的コマンドを提供する
- README は説明書ではなく導線として再構成する
- スクリプトは成功時だけでなく、失敗時のメッセージ品質を重視する
- `bootstrap` や `doctor` の実装に追加パッケージが必要な場合は、都度導入を許容する
- 追加パッケージは用途が明確で、CLI 実装やクロスプラットフォーム対応に寄与するものに限定する
- 現時点の主対象は macOS / Linux としつつ、シェル依存やパス依存を減らして Windows 対応余地を残す

## 設計制約

- 初期実装では Windows の動作保証までは行わない
- ただしセットアップ手順とスクリプト設計は Windows 展開を妨げないことを条件にする
- 可能な限り Bash 専用実装ではなく Bun / TypeScript ベースへ寄せる
- パス区切り、実行ファイル名、`docker compose` 呼び出し差異、改行コード差異を吸収しやすい構成にする
- OS 判定で分岐する場合も、分岐理由をコードとドキュメントに残す

## フェーズ 1: 最小導線の定義と README 再構成

### 目的

初見ユーザーが読むべき内容を最小化し、最初の 10 分で迷わない状態を作る。

### 作業

- `README.md` 冒頭を「5分で最小起動」に差し替える
- 構成を `最小構成` `標準構成` `フル構成` に分ける
- 必須依存を明示する
  - Bun
  - Docker
  - Python 3.10+
- 任意コンポーネントを明示する
  - local-llm
  - Monitor
  - cloud review
- 現時点の正式サポート OS と、将来考慮する OS を明記する
- 「最小起動でできること」を短く列挙する
- よくある失敗を 3 件程度 README に入れる

### 対象ファイル

- `README.md`
- `CONTRIBUTING.md`
- 必要に応じて `docs/configuration.md`

### 完了条件

- README 冒頭だけで最小セットアップが完結する
- 最小構成に不要な情報が先頭導線へ混ざっていない
- `git clone` 直後に読むファイルが事実上 `README.md` のみになる

## フェーズ 2: 設定ファイルの簡素化

### 目的

`.env` の判断コストを下げ、必須設定と任意設定を分離する。

### 作業

- `.env.example` を用途別に分割する
  - `.env.minimal`
  - `.env.local-llm`
  - `.env.cloud-review`
- 既存 `.env.example` は互換維持のために残す場合でも、用途別テンプレートへの案内役にする
- 各テンプレートに「この構成で必要な設定だけ」を残す
- `docs/configuration.md` に「どの構成でどの env を使うか」の表を追加する
- Windows でも解釈しやすい env 命名とパス指定ルールを採用する

### 対象ファイル

- `.env.example`
- 新規 `.env.minimal`
- 新規 `.env.local-llm`
- 新規 `.env.cloud-review`
- `docs/configuration.md`

### 完了条件

- 最小構成では DB 周りの設定だけで開始できる
- Bedrock や OpenAI の設定が、ローカル最小起動の邪魔をしない
- 新規利用者が「この値は今必要か」を迷わず判断できる

## フェーズ 3: `bootstrap` コマンドの追加

### 目的

`git clone` 後の主要セットアップを単一コマンドで実行できるようにする。

### 作業

- `bun run bootstrap` を追加する
- 以下を段階実行する
  - Bun 依存インストール確認
  - `.env.minimal` から `.env` 生成
  - Python サービスセットアップ
  - Docker DB 起動
  - DB 初期化
- 途中失敗時に、失敗したステップ名と再実行方法を表示する
- 既存 `monorepo:setup` は内部サブステップとして再利用する
- 実装に `execa` などの補助パッケージが必要なら追加する
- shell script に依存しすぎず、Windows 展開しやすい Bun 実装を優先する

### 実装メモ

- まずは Bun スクリプトで実装し、OS 依存を増やしすぎない
- `docker compose` と `docker-compose` の差異は吸収する
- `.env` が既に存在する場合は上書きせず、明示的に通知する
- `python3` と `python`、仮想環境パス、実行ファイル拡張子の差異を吸収できる余地を持たせる

### 対象ファイル

- `package.json`
- 新規 `scripts/bootstrap.ts` または `scripts/bootstrap.sh`
- 必要に応じて `scripts/setup-services.sh`

### 完了条件

- fresh clone 後に `bun run bootstrap` で最小構成が揃う
- `.env` の有無で危険な上書きが起きない
- ステップ単位の失敗メッセージが読める

## フェーズ 4: `doctor` コマンドの追加

### 目的

導入失敗時に、利用者が現状を自力で切り分けできるようにする。

### 作業

- `bun run doctor` を追加する
- 次を検査対象にする
  - Bun の存在
  - Python の存在
  - Docker の存在
  - PostgreSQL コンテナの起動状態
  - `DATABASE_URL` 接続
  - `GNOSIS_EMBED_COMMAND` の存在
  - local-llm の基本疎通
- 各項目を `OK/WARN/FAIL` で出す
- FAIL の場合は、その場で打てる修正コマンドを表示する
- 実装に表示整形やコマンド実行補助のパッケージが必要なら追加する
- Windows では未サポート項目を `WARN` 扱いで説明できる構造にする

### 対象ファイル

- `package.json`
- 新規 `scripts/doctor.ts`
- 必要に応じて既存の health check ロジック

### 完了条件

- 利用者が issue を開く前に最低限の自己診断ができる
- 主要な失敗パターンが `doctor` の出力で説明される

## フェーズ 5: 導入確認用 smoke コマンドの追加

### 目的

「セットアップできたか」を判定する共通の成功ラインを作る。

### 作業

- `bun run onboarding:smoke` を追加する
- 次の軽量チェックを実行する
  - DB 接続
  - vector extension の存在確認
  - seed 済み確認
  - MCP サーバーの最小起動確認
- 必要なら local-llm 非依存で完結する内容に限定する

### 対象ファイル

- `package.json`
- 新規 `scripts/onboarding-smoke.ts`

### 完了条件

- 利用者が「動いている」と判断するための単一コマンドが存在する
- local-llm 未導入でも最小構成の確認ができる

## フェーズ 6: fresh clone CI の追加

### 目的

README の導入手順が将来壊れても、すぐ検知できるようにする。

### 作業

- GitHub Actions を追加する
- クリーン環境で以下を検証する
  - `bun install`
  - `bun run bootstrap`
  - `bun run doctor`
  - `bun run onboarding:smoke`
- Docker が使える CI 構成にする
- README の手順と CI の手順を一致させる
- 可能なら将来の Windows CI 追加を見越して workflow を分割しやすい構造にする

### 対象ファイル

- 新規 `.github/workflows/onboarding.yml`
- 必要に応じて README

### 完了条件

- fresh clone の最小導入手順が CI で常時検証される
- README と CI の乖離が起きにくい

## 推奨実施順

1. フェーズ 1
2. フェーズ 2
3. フェーズ 3
4. フェーズ 4
5. フェーズ 5
6. フェーズ 6

この順序にする理由は、導線と前提を固める前に自動化を進めると、後からコマンド仕様がぶれやすいため。

## 受け入れ指標

- 新規利用者が `README` を 1 ファイル読むだけで最小起動できる
- 導入コマンドが 3 本以内に収まる
  - `bootstrap`
  - `doctor`
  - `onboarding:smoke`
- 初回導入時の「何が必須か分からない」という問い合わせが減る
- CI 上で fresh clone 導線が継続的に成功する
- CLI 実装のために必要な追加パッケージが、責務を説明できる形で管理される
- Windows 未対応のままでも、Windows 対応を阻害する実装が増えない

## リスクと対策

### `bootstrap` が肥大化する

- 対策: 最小構成を対象に限定し、フル構成は別コマンドへ分離する

### local-llm 依存を早く入れすぎて失敗率が上がる

- 対策: onboarding 導線から local-llm を切り離し、任意フェーズに後置する

### `.env` テンプレート分割で既存利用者が混乱する

- 対策: 旧 `.env.example` は当面残し、README で新テンプレートを優先案内する

### 実装が Bash 前提に寄りすぎて Windows 展開しづらくなる

- 対策: 新規導線は Bun スクリプト中心に寄せ、既存 shell script は段階的に内部化する

### 補助実装のための追加パッケージが増えすぎる

- 対策: 標準ライブラリで足りる範囲を優先し、追加時は用途を明確にして最小限に留める

## 直近の着手候補

最初の PR は、以下だけに絞るのが妥当。

1. README の最短導線化
2. `.env` テンプレート分割
3. `bootstrap` の追加

この 3 点だけでも、`git clone` 直後の体験はかなり改善する。
