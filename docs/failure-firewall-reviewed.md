## 1. 概要

`docs/failure-firewall.md` に記載された実装計画について、Astmend の改良によって解決される「構造解析の精度」以外の、運用・アーキテクチャ・人間系に関する懸念事項を以下にまとめる。

---

## 2. 運用上の懸念点

### 2.1 ナレッジのコールドスタート問題
AST 解析が強力になっても、肝心の「Golden Path（成功の型）」が登録されていなければ機能しない。
- **リスク**: 初期導入時に開発者が手動で Golden Path を書くコストが高く、普及が遅れる。
- **対策案**: 
    - 既存の「正常に動作しているコード」から LLM が Golden Path を逆引きで自動生成するボトムアップ機能。
    - TanStack Query や Zod などの主要ライブラリに対する「標準 Golden Path セット」のプリセット提供。

### 2.2 フレームワークの進化とナレッジの陳腐化
技術スタックのアップデート（例：React Query v4 から v5 への移行）により、既存の Golden Path が「古い推奨」に変わってしまう。
- **リスク**: 古い Golden Path が新しい（より良い）実装方法を「逸脱」として誤検知し続ける。
- **対策案**: 
    - Golden Path への「有効なフレームワークバージョン」メタデータの付与。
    - `record_task_note` または scheduled synthesis で「この実装は古い Golden Path を意図的に更新したものである」と判断された場合、ナレッジを自動更新（Deprecated 化）するフロー。

### 2.3 開発者の「検知疲れ」と False Positive への耐性
ブロック条件が厳しすぎると、開発者は Failure Firewall を「邪魔な存在」と認識し、無効化または無視するようになる。
- **リスク**: 誤検知（False Positive）が数回続くことで、真に重大な警告も無視されるようになる。
- **対策案**: 
    - 初期段階では `error` (Block) を極めて限定的にし、`warning` 中心で運用する。
    - 「この警告は妥当ではない」というフィードバックをワンクリックで行える UI と、そのフィードバックが即座に検索スコア（penalty）に反映される仕組み。

## 3. アーキテクチャ・技術的懸念点

### 3.1 実行時パフォーマンスと開発体験（DX）
AST 解析、ベクトル検索、LLM 判定のフルセットを `dev` サイクル（pre-commit hook 等）で走らせるのは重すぎる可能性がある。
- **リスク**: `git commit` やレビュー依頼のたびに 10 秒以上の待機が発生し、開発リズムが削削がれる。
- **対策案**: 
    - バックグラウンド実行（daemon モード）による非同期チェック。
    - 変更範囲が小さい場合は AST 解析をスキップし、リスク信号のキーワードマッチのみで一次フィルターを行う多段構成。

### 3.2 マルチ言語プロジェクトでの格差
Astmend は TypeScript に特化しているが、プロジェクトには CSS、SQL、YAML、Python などが混在する。
- **リスク**: TypeScript 以外では検知精度が著しく落ち、プロジェクト全体の品質担保にムラができる。
- **対策案**: 
    - 言語ごとに解析エンジンの「信頼度スコア」を持ち、信頼度が低い言語では LLM へのコンテキスト注入を厚くするなどの動的調整。

## 4. 人間・プロセスに関する懸念点

### 4.1 ゴールデンパスの合意形成（ガバナンス）
「何が正解か」は開発者によって好みが分かれる場合がある。
- **リスク**: 特定の個人の「こだわり」が Golden Path として登録され、他のメンバーに強制されることによるチーム内の不和。
- **対策案**: 
    - Golden Path の `active` 昇格に「チームメンバーによる承認」ステップを導入する。
    - プロジェクト全体、モジュール、個人、という階層的なスコープ管理。

### 4.2 セキュリティとプライバシー
「失敗体験」に機密性の高いコードや PII（個人情報）が含まれる可能性がある。
- **リスク**: 失敗ログがベクトルデータベースに保存され、不適切に他のエージェントや LLM に共有される。
- **対策案**: 
    - ナレッジ保存時の自動アノニマイズ（匿名化）。
    - 失敗体験を「コードそのまま」ではなく「抽象化されたパターンと構造」に変換してから永続化する。

---

## 5. 結論

Failure Firewall の成功は、ツールの解析能力（Astmend）と同じくらい、**「いかに低摩擦でナレッジを最新に保ち、チームの合意を反映させるか」**という運用設計に依存する。実装においては、解析エンジンの開発と並行して、これらの人間系プロセスの自動化・支援機能を組み込むことが強く推奨される。
